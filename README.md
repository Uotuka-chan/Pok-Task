# TaskScope

Webアプリケーション開発実習の最終成果物です。SPEC駆動開発の考え方に基づき、実装範囲を小さく定義し、DB・API・UIが一貫してつながることを重視しました。

## アプリケーション概要

TaskScopeは、学習や提出準備で必要なタスクを確認するためのタスク一覧アプリです。対象ユーザーは、課題や作業予定を見失いやすい学生です。SQLiteに保存したタスクをNext.jsのAPI Routeで取得し、Reactコンポーネントで一覧表示します。今回はMVPとして、登録・更新・削除は実装せず、seedで用意した初期データを安定して表示することを目的にしています。

## 使用技術

- Next.js
- TypeScript
- Tailwind CSS
- Prisma
- SQLite

## セットアップ手順

```bash
npm install
cp .env.example .env
npx prisma migrate dev --name init
npx prisma db seed
npm run dev
```

確認URL:

- 画面: http://localhost:3000
- API: http://localhost:3000/api/tasks

## SPEC 1：画面仕様

| 項目 | 内容 |
|---|---|
| 画面名 | タスク一覧画面 |
| URL | `/` |
| 目的 | DBに保存されたタスクを一覧表示する |
| 認証 | 今回はなし |

### 表示する情報

| 項目 | 内容 |
|---|---|
| タスクタイトル | タスク名を表示する |
| 完了状態 | 完了／未完了をバッジで表示する |
| 期限日 | 期限日がある場合は月/日形式で表示する |
| 集計情報 | 総タスク数と完了済み件数を表示する |

### 画面状態

| 状態 | 表示内容 |
|---|---|
| loading | 「読み込み中...」を表示する |
| success | タスク一覧を表示する |
| empty | 「タスクはありません」を表示する |
| error | 「データの取得に失敗しました」を表示する |

|---|---|---|---|

リレーション: `User 1 --- n Task`

## SPEC 3：API定義

| Method | Endpoint | 内容 |
|---|---|---|
| GET | `/api/tasks` | userId=1のタスク一覧を取得する |

### GET /api/tasks

正常系:

- DBからuserId=1のタスクを取得する
- 作成日の降順で返す
- `id`, `title`, `isCompleted`, `dueDate`, `createdAt` をJSONで返す

非正常系:

- DBアクセスに失敗した場合、HTTP 500と汎用エラーメッセージを返す
- サーバー側では `console.error` により原因調査用ログを残す

## SPEC 4：UIコンポーネント定義

| コンポーネント | 役割 |
|---|---|
| `TaskCard` | 1件のタスク情報をカード形式で表示する |
| `TaskList` | タスク配列を受け取り、`TaskCard`を繰り返し表示する |
| `StateMessage` | loading / empty / error の状態表示を共通化する |

## 機能一覧

| 機能ID | 機能名 | 正常系 | 非正常系 | 対象外 |
|---|---|---|---|---|
| F-01 | タスク一覧表示 | seedデータを一覧表示する | DB取得失敗時にエラー表示 | ページネーション |
| F-02 | 完了状態表示 | 完了／未完了をバッジ表示する | 不正データは表示対象外 | 完了状態の更新 |

## フリー記述レポート

本アプリでは、授業で学んだSPEC駆動開発を意識し、最初に「何を作るか」と「今回は作らないこと」を明確に分けた。自由に機能を増やすと、登録・更新・削除・ログインなどの実装範囲が広がり、SPECと実装の整合性が崩れる可能性がある。そのため今回は、タスク一覧表示、完了状態表示、状態別UI表示の3機能に絞った。これにより、seedデータ、Prisma Schema、API Route、Reactコンポーネント、テストケースが同じ仕様を指すように整理できた。

工夫した点は、UIを`TaskCard`、`TaskList`、`StateMessage`に分割したことである。1件のタスク表示、一覧表示、状態表示の責務を分けることで、画面全体の見通しが良くなり、修正もしやすくなった。また、API側ではDB取得失敗時にHTTP 500を返し、画面側ではユーザーに汎用的なエラーメッセージを表示するようにした。詳細なエラーはサーバーログに残すことで、利用者に不要な内部情報を見せない設計にしている。

今後の改善点としては、タスク登録、完了状態の更新、削除、検索機能を追加したい。ただし、追加する場合も先にSPECとテストケースを更新し、実装だけが先行しないようにする必要がある。今回の成果物は小規模だが、DB、API、UIがつながり、第三者がseedを実行して動作確認できる状態を目指した。
